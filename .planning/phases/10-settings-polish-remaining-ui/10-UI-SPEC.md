## Spacing Scale

All values are multiples of 4. Tailwind v4 logical-property utilities apply throughout (`ps-`, `pe-`, `ms-`, `me-` — never `pl-`, `pr-`, `ml-`, `mr-`).

| Token | Value | Tailwind Utility | Usage |
|-------|-------|-----------------|-------|
| xs | 4px | `gap-1`, `p-1` | Icon gaps, inline badge padding |
| sm | 8px | `gap-2`, `p-2` | Compact spacing within list items |
| md | 16px | `px-4`, `py-4` | Default button padding, input padding, card interior |
| lg | 24px | `px-5`, `pt-6` | Screen horizontal padding (matches existing screens: `px-5 pt-6`) |
| xl | 32px | `space-y-8` | Major section gaps within a screen |
| 2xl | 48px | `py-12` | Empty state vertical centering, full-screen center regions |
| 3xl | 64px | `pb-28` | Bottom padding on all scrollable content areas to clear the fixed nav bar |

Exceptions:
- Touch targets on nav tab items: minimum 44px tall (`py-3` + icon + label = ~48px). Source: `AppBottomNav.tsx` confirms `py-3` with 22px icon + text = ~48px total. Compliant.
- `pb-28` (112px) on all scrollable screen content: this exceeds the scale but is a confirmed pattern (ProgressScreen, NutritionDashboard, CompanionChat) required to clear the 64px fixed bottom nav plus safe-area inset.
- Safe-area inset padding: `.pb-safe` = `env(safe-area-inset-bottom, 16px)` applied to the nav bar itself. Source: `globals.css`.

## Typography

Font sizes are in pixels. All body copy uses `font-family-sans` (system stack, Persian-aware). No web font loading — no FOUT risk.

| Role | Size | Weight | Line Height | Tailwind Classes | Usage |
|------|------|--------|-------------|-----------------|-------|
| Body | 15px | 400 (regular) | 1.6 | `text-sm leading-relaxed` (Tailwind `sm` = 14px; use `text-[15px]` where precision required) | Form labels, list item descriptions, card body copy, error messages, settings row values |
| Label / UI | 14px | 400 (regular) | 1.4 | `text-xs` (Tailwind `xs` = 12px; `text-sm` for 14px) | Nav tab labels, badge text, secondary annotations, setting row sublabels |
| Heading | 20px | 700 (bold) | 1.3 | `text-xl font-bold` | Screen titles (`settings.title`), section headers within settings, empty state headings |
| Display | 28px | 700 (bold) | 1.2 | `text-2xl font-bold` | Not used in Phase 10 — reserved for splash/dashboard |

Declared font weights: **400 (regular)** and **700 (bold)**.

Notes:
- `UI-STYLE-04` mandates minimum 15px body, ≥1.6 line height, spacious padding. This contract enforces it.
- Persian numerals are handled by the browser rendering the Unicode range — no special font weight change needed.
- Empty state body copy uses `text-sm text-ink-2 leading-relaxed` matching ProgressScreen.tsx pattern.

### Standardized State Patterns

All async screens must implement these three patterns exactly. They are already established in `ProgressScreen.tsx` and `AuthGuard.tsx` — replicate, do not reinvent.

**Loading state:**
```tsx
if (loading) {
  return (
    <div className="flex-1 flex items-center justify-center min-h-[60vh]">
      <div
        role="status"
        aria-label={dict.common.loading}
        className="w-8 h-8 rounded-full border-2 border-brand border-t-transparent animate-spin"
      />
    </div>
  )
}
```

**Empty state:**
```tsx
<div className="flex-1 overflow-y-auto px-5 pt-6 pb-28">
  <div className="rounded-2xl bg-elevated p-6 shadow-sm text-center space-y-4">
    <div className="mx-auto w-20 h-20 rounded-full bg-brand-muted flex items-center justify-center">
      <span className="text-3xl">{contextIcon}</span>
    </div>
    <h2 className="text-xl font-bold text-ink">{dict.section.emptyTitle}</h2>
    <p className="text-sm text-ink-2 leading-relaxed">{dict.section.emptyDesc}</p>
    <button type="button" onClick={primaryAction}
      className="w-full py-4 rounded-2xl bg-brand text-elevated font-bold text-sm">
      {dict.section.emptyCta}
    </button>
  </div>
</div>
```

**Error state:**
```tsx
if (loadError) {
  return (
    <div className="flex-1 px-5 pt-6 pb-28">
      <div className="rounded-2xl bg-elevated p-6 shadow-sm text-center space-y-3">
        <p className="text-sm text-error">{loadError}</p>
        <button type="button" onClick={() => void reload()}
          className="px-4 py-2 rounded-2xl bg-brand text-elevated font-bold text-sm">
          {dict.common.retry}
        </button>
      </div>
    </div>
  )
}
```

**UNAUTHORIZED catch pattern (all screens):**
```tsx
} catch (err) {
  if (err instanceof Error && err.message === 'UNAUTHORIZED') {
    router.replace(`/${locale}/login`)
    return
  }
  setLoadError(err instanceof Error ? err.message : dict.section.errLoadFailed)
}
```

### Destructive Actions

Phase 10 has exactly one destructive action: **logout**.

Interaction contract for logout:
1. User taps "خروج از حساب" / "Log out" button
2. An inline confirmation section expands below the button (no modal, no separate screen — inline state using React `useState`)
3. User sees two buttons: "بله، خارج شو" (confirm, `text-error`) and "نه، بماندم" (cancel, `text-ink-2`)
4. Confirming calls `POST /api/v1/auth/logout`, clears JWT cookie, redirects to `/{locale}/login`
5. Cancelling collapses the confirmation section

No other destructive actions exist in Phase 10. Settings profile is read-only. Language switching is reversible.

## Interaction Contracts

### Language Switching

1. User opens Settings screen → taps the Language row
2. Browser navigates to `/{locale}/settings/language`
3. Language Selector screen renders all 3 locales; active locale is highlighted with `bg-brand-muted text-brand font-bold`
4. User taps a different locale
5. Client writes `NEXT_LOCALE={newLocale}` cookie (path=/, max-age=1 year, SameSite=Lax)
6. Client calls `router.push('/{newLocale}/settings/language')`
7. Server re-renders with new `lang` and `dir` on `<html>` — no flicker
8. IMPORTANT: Do NOT call `document.documentElement.dir = '...'` anywhere. Cookie + server render handles direction.
9. Backend `PATCH /api/v1/settings/language` is called as a fire-and-forget after the cookie write (errors are silently ignored — cookie is the source of truth for UX)

### Bottom Nav Settings Tab Enable

The `AppBottomNav` settings tab is currently `disabled: true`. After the settings route exists, patch exactly:
- `disabled: true` → `disabled: false`
- `active: false` → `active: isActive('settings')`

The tab is not enabled until the page file at `app/[lang]/settings/page.tsx` exists and returns 200.

### Screen Content Scroll Pattern

Every content area on every screen must follow this pattern exactly:
```
<div className="flex-1 overflow-y-auto px-5 pt-6 pb-28 space-y-6">
```
- `flex-1` — fills remaining height in the flex column `app-container`
- `overflow-y-auto` — scrollable content
- `px-5` — 20px horizontal padding (logical, works RTL)
- `pt-6` — 24px top padding
- `pb-28` — 112px bottom padding clears the 64px fixed nav + safe area

### RTL Compliance Contract

All new and patched components must pass this gate before review:

1. Zero `pl-*`, `pr-*`, `ml-*`, `mr-*` Tailwind classes — use `ps-*`, `pe-*`, `ms-*`, `me-*`
2. All directional icons (chevrons, back arrows) use `getIconFlipClass(locale)` from `lib/direction.ts`
3. No `dir="rtl"` or `dir="ltr"` hard-coded on any component — direction is only on `<html>` from root layout
4. Text alignment is inherited from `<html dir>` — no `text-right` or `text-left` in component files

Automated audit command (run from `frontend/` directory — any output is a violation):
```bash
grep -rn "pl-\|pr-\|ml-\|mr-" src/components/ src/app/ --include="*.tsx"
```

## Backend Contract

### New Endpoint Required

| Method | Path | Auth | Purpose |
|--------|------|------|---------|
| PATCH | `/api/v1/settings/language` | `Depends(get_current_user)` | Persist language preference to `UserLanguagePreference` table |

Request schema:
```python
class UpdateLanguageRequest(BaseModel):
    language_code: Literal["fa", "en", "ar"]
```

Response schema:
```python
class LanguagePreferenceResponse(BaseModel):
    language_code: str
    updated_at: datetime
```

No new `GET` endpoint needed for settings profile — reuse existing `GET /api/v1/auth/me`.

The backend call is fire-and-forget from the client. UX is not blocked by its success or failure. The `NEXT_LOCALE` cookie is the canonical client-side source of truth.

## Anti-Patterns

These are explicitly prohibited in all Phase 10 work:

| Anti-Pattern | Correct Pattern |
|-------------|----------------|
| `document.documentElement.dir = 'rtl'` in React component | Cookie write + `router.push()` — server re-renders `<html dir>` |
| `pl-4`, `pr-4`, `ml-4`, `mr-4` Tailwind classes | `ps-4`, `pe-4`, `ms-4`, `me-4` |
| Hard-coded strings: `"تنظیمات"` or `"Settings"` in JSX | `dict.settings.title` |
| `err.message` displayed raw to user | Map to `dict.section.errLoadFailed` or `dict.errors.generic` |
| `<div style={{ paddingLeft: '16px' }}>` | `className="ps-4"` |
| Modal library for logout confirmation | Inline React state — `const [confirmLogout, setConfirmLogout] = useState(false)` |
| `text-right` or `text-left` in components | Remove — direction inherited from `<html dir>` |
| Missing `pb-28` on scrollable content | `<div className="flex-1 overflow-y-auto px-5 pt-6 pb-28">` |
| Missing `app-container` on new page | Wrap every page root in `<div className="app-container">` |
| Enabling settings nav tab before route exists | Create `app/[lang]/settings/page.tsx` first, verify 200, then patch nav |
